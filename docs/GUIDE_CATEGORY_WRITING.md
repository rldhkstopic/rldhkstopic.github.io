# 카테고리별 작성 가이드

이 문서는 블로그 포스트의 카테고리별 작성 스타일과 가이드라인을 설명합니다.

## 📋 카테고리 개요

현재 지원되는 카테고리:
- **dev**: 개발 관련 글 (코드, 트러블슈팅, 기술 스택 등)
- **study**: 학습/공부 관련 글 (개념 정리, 학습 노트 등)
- **daily**: 일상/회고 글 (개인 경험, 일기 형식)
- **document/essay**: 분석/에세이 글 (데이터 분석, 인사이트 등)

## 📝 Daily (일상/회고)

### 특징
- **1인칭 시점**: '나', '내가', '나는' 등 개인의 관점에서 서술
- **감정 묘사 필수**: 단순 사실 나열이 아닌, 그때 느꼈던 감정을 구체적으로 서술
- **현장감**: 추상적 표현 대신 구체적인 묘사 사용
- **개인 경험 중심**: 뉴스 기사나 보고서 같은 객관적 서술 지양
- **자연스러운 구어체**: 일상 대화하듯 자연스럽고 편안한 문체

### 작성 규칙

1. **1인칭 시점 유지**
   - ✅ "나는 드론쇼를 보러 갔다"
   - ❌ "드론쇼는 많은 사람들이 관람했다" (3인칭 관찰자 시점)

2. **감정 묘사 강화**
   - ✅ "짜증이 밀려왔다", "기쁨이 샘솟았다", "당황했다"
   - ❌ "사람이 많았다", "이벤트가 진행되었다" (감정 없는 사실만)

3. **구체적인 묘사**
   - ✅ "지하철역까지 가는 데만 30분이 걸려서 발이 퉁퉁 부었다"
   - ❌ "대중교통 시스템이 마비되었다" (추상적)

4. **구조**
   - [상황] 어디서 무엇을 겪었나
   - [행동] 무슨 일이 있었고 어떻게 했나
   - [회고] 무엇을 느꼈고 다음엔 어떻게 할 것인가

5. **문체 (매우 중요)**
   - **자연스러운 구어체 사용**: "~다." 문체보다는 "~했다", "~였다", "~했었다" 같은 자연스러운 과거형 사용
   - **짧고 간결한 문장**: 길고 복잡한 문장보다는 짧고 명확한 문장 선호
   - **일상 대화 톤**: 친구에게 말하거나 일기장에 쓰는 듯한 편안한 톤
   - **구어체 어미 허용**: "~했음", "~했었는데", "~했더니", "~해서" 등 자연스러운 구어체 표현 사용
   - **과도한 격식 지양**: "~임을 확인했다", "~인 것으로 보인다" 같은 딱딱한 표현 피하기
   - ✅ 좋은 예: "어제 10시까지 야근해서 그런지 아침에 몸이 무거웠다. 9시 반쯤 되어서야 집에서 출발했다."
   - ❌ 나쁜 예: "어제 10시까지 야근한 결과, 아침에 몸이 무거웠음을 확인했다. 9시 반이 되어서야 집에서 출발할 수 있었다."

6. **민감 정보 필터링**
   - **실명 제거**: 사람 이름은 "A 선임", "B 선임", "수석님" 등으로 일반화
   - **회사/프로젝트명 일반화**: 구체적인 회사명이나 프로젝트명은 일반적인 표현으로 변경
   - **업무 세부사항 완화**: 민감한 업무 내용은 핵심만 남기고 구체적인 수치나 명칭은 생략
   - ✅ 좋은 예: "수석님께 하드웨어 조립체 견적 관련해서 여쭤봤다"
   - ❌ 나쁜 예: "김수석님께 포선조립체 견적 관련해서 여쭤봤다" (실명 포함)

### 금지 사항
- "본고는", "분석한다", "시사한다", "논한다" 같은 딱딱한 논문조 표현
- 3인칭 관찰자 시점
- 뉴스 기사나 보고서 같은 객관적 서술
- 감정 없이 팩트만 나열

### 예시

**좋은 예 (자연스러운 구어체):**
```
어제 10시까지 야근해서 그런지 아침에 몸이 무거웠다. 
9시 반쯤 되어서야 집에서 출발했다. 
출근하는 내내 어제 일이 머릿속을 맴돌았다. 
시스템 확정 짓는 과정에서 질문을 받았는데, 
내가 제어하는 모듈인데도 구체적인 내용을 제대로 이해하지 못해 당황했었다. 
특히 파일 구조나 제어 방식에 대해 제대로 파악하지 못하고 있었던 게 
그대로 드러난 것 같아 마음이 불편했다.
```

**나쁜 예 (너무 격식있는 문어체):**
```
어제 10시까지 야근한 결과, 아침에 몸이 무거웠음을 확인했다. 
9시 반이 되어서야 집에서 출발할 수 있었다. 
출근하는 동안 어제의 업무가 머릿속을 맴돌았다. 
시스템 확정 과정에서 질문을 받았는데, 
내가 제어하는 모듈임에도 불구하고 구체적인 내용을 제대로 이해하지 못하여 당황했다. 
특히 파일 구조나 제어 방식에 대해 제대로 파악하지 못하고 있었던 것이 
그대로 드러난 것으로 보여 마음이 불편했다.
```

## 💻 Dev (개발)

### 특징
- **기술 중심**: 코드, 트러블슈팅, 기술 스택에 집중
- **논리적 흐름**: 단계별로 명료하게 정리
- **실용성**: 실제 사용 가능한 해결책 제시
- **경험 기반**: 직접 경험한 문제 해결 과정을 중심으로 서술

### 기술 블로그를 쉽게 쓰는 방법 3가지

#### 1. 주제 의식을 버리고 소재 의식으로 쓰자

**주제 의식**이란 자존감, 자본주의 같은 추상적 가치를 기반으로 하는 글쓰기다. 반면 **소재 의식**은 특정한 상황에 대한 자기만의 관점이나 생각, 해결 방안을 뜻한다.

소재 의식은 독자와 상관없이 특정 상황에 맞닥뜨렸을 때부터 해결할 때까지의 일을 정리해서 작성하면 된다.

**작성 패턴:**
- `~로 인해 ~상황이 발생했고 그 상황을 ~방식으로 해결한 내용`
- 문제 해결 과정을 담담하게 정리
- 기술 블로그는 일상을 다룬 수필이나 에피소드와 비슷하다

**예시:**
```
VHDL에서 wait 문은 시뮬레이션에서는 매우 편하지만, 합성에서는 제한이 크다. 
Vivado 합성에서 wait statement not supported for synthesis 오류가 발생했다. 
이 문제를 해결하기 위해 process 문을 사용하여 클럭 기반 동기화로 변경했다.
```

#### 2. 독자 수준이 아니라 자기 수준으로 쓰자

기술 블로그를 독자 수준에 맞춰 쉽게 쓰는 것은 굉장히 어렵다. 무작정 쉽게 쓴다고 해서 독자의 활용도가 높아지는 것도 아니다.

**작성 원칙:**
- 독자 수준이 아니라 작성자 수준으로 쓰기
- 어려운 용어를 일부러 해석해 풀어쓰거나 쉬운 용어로 바꿀 필요 없음
- 필요하다면 해당 용어에 대한 설명이 담긴 링크를 제공
- 해설을 늘어놓다 보면 정작 할 말을 못 하게 됨

**예시:**
```
VHDL의 wait 문은 합성 불가능하다. 대신 process 문과 클럭 신호를 사용해야 한다.
[참고: VHDL Process 문](링크)
```

#### 3. 재미있게 글을 쓰자

좋은 기술 블로그는 개발자의 경험에서 우러나오는 내용을 적절한 글쓰기 기교로 녹여낸 것이다. 재미가 없다면 잘 읽히지 않고, 스스로도 흥미를 잃게 된다.

**작성 기교:**
- 개발자의 경험을 구체적으로 서술
- 독자가 저자의 경험을 공감하고 끄덕이게 만들기
- 적절한 글쓰기 기교 활용

### 글의 종류별로 목차 잡는 법 - 저, 술, 편, 집

개발자가 기술 블로그에 쓰는 글은 크게 **저(著), 술(述), 편(編), 집(輯)** 4가지로 나뉜다.

| 구분 | 내용 | 종류 |
|------|------|------|
| **저** | 직접 경험하고 실험한 과정이나 결과 | 개발기, 도입기, 적용기 |
| **술** | 어떤 것을 분석하여 의미를 풀이하고 해석한 것 | 기술 소개, 용어 분석, 에러 해결 방법 등 |
| **편** | 산만하고 복잡한 자료를 편집해 질서를 부여한 것 | 프로그램 설치/설정 방법, 튜토리얼, 세미나 후기, 책 리뷰 |
| **집** | 여러 사람의 견해나 흩어진 자료를 한데 모아 정리한 것 | 명령어 모음, 팁 등 |

#### 저(著): 개발기는 목차를 잘 잡아서 본문부터 쓰자

개발 과정과 결과를 쓴 개발기가 저에 해당한다.

**작성 방법:**
1. **목차 구성**: 성공한 루트와 실패한 루트를 구별
   - 개발 모델(최종 루트)을 기준으로 목차 잡기
   - 개발 과정에서 발견한 문제와 해결 방법(경험 루트)을 별도 섹션으로 정리
2. **작성 순서**: 머리말보다 본문을 먼저 쓰기
   - 개발자가 제일 잘 쓸 수 있는 본문을 먼저 작성
   - 머리말은 마지막에 블로그에 올릴 때 생각나는 대로 간략히 작성

**예시 목차:**
```
1. 문제 상황
2. 해결 과정
   2.1 첫 번째 시도 (실패)
   2.2 두 번째 시도 (성공)
3. 최종 해결 방법
4. 참고 사항
```

#### 술(述): 원전을 비교하고 실험해 풀이해서 쓰자

어떤 것을 분석해 의미를 풀이하고 해석한 것이 술이다. 새로운 기술을 자세하게 설명한 것, 비슷한 용어를 풀이한 것, 에러 해결법이 해당한다.

**작성 방법:**
1. **원전 활용**: 개발에도 원서, 원문, 레퍼런스가 있다
   - 원전의 내용을 먼저 쓰고 비교한 내용을 추가
   - 자기 생각이나 분석, 해설을 덧붙이기
2. **비교 분석**: 원전의 두 용어나 대상을 비교하면서 풀이
3. **실험 정리**: 원전의 내용을 직접 실험해 비교한 뒤 그 내용을 정리

**예시 목차:**
```
1. HTTP 개요
2. GET 메서드
3. POST 메서드
4. GET과 POST의 차이 (비교 분석)
5. 참고
```

#### 편(編): 순서를 요약하여 쓰자

산만하고 복잡한 자료를 편집해 질서를 부여한 것이 편이다.

**작성 방법:**
1. **시간 순서로 나열**: 일단 먼저 시간 순으로 내용을 나열
2. **단계로 묶기**: 나열한 목록을 단계로 나눠서 목차로 만들기
3. **요약**: 각 단계에 속한 내용을 간략하게 요약

**예시:**
```
1. 환경 설정
   1.1 Node.js 설치
   1.2 패키지 설치
2. 프로젝트 구성
   2.1 디렉토리 구조
   2.2 설정 파일 작성
3. 실행 및 테스트
```

#### 집(輯): 글쓰기가 두렵다면 자료를 모아 핵심을 엮어서 쓰자

여러 사람의 견해나 흩어진 자료를 한데 모아 정리하는 것이 집이다.

**작성 방법:**
- 내용을 많이 쓰는 것이 아니라 핵심만 간결하게 정리
- 글 전체가 길지 않고 내용이 각각 나열되어 있음
- 본인이 경험에서 터득한 것을 핵심만 정리해서 나열해도 좋음

**예시:**
```
## VHDL 코딩 팁

1. wait 문은 합성 불가능하므로 process 문 사용
2. std_logic_vector는 명시적 타입 변환 필요
3. ...
```

### 작성 규칙

1. **코드 흐름과 트러블슈팅 과정을 단계별로 명료하게 정리**
   - 문제 인식 → 원인 분석 → 해결 방법 → 검증

2. **코드/에러 메시지는 필요한 최소한만 남기되 설명은 한국어로 풀어서 작성**

3. **구조**
   - [현상/문제 인식] → [데이터/근거 분석] → [전문가 의견 대조] → [인사이트 도출]

4. **문체**
   - "~다."로 끝나는 건조하고 분석적인 문체
   - 감정 배제, 이모지 금지

5. **목차 작성**
   - 저(개발기): 성공한 루트를 기준으로 목차 구성, 본문부터 작성
   - 술(분석): 원전 내용을 먼저 쓰고 비교 분석 추가
   - 편(정리): 시간 순서로 나열 후 단계로 묶어 요약
   - 집(모음): 핵심만 간결하게 정리하여 나열

### 금지 사항
- "안녕하세요", "반갑습니다" 같은 인사
- "결론적으로", "요약하자면" 같은 접속사
- "매우", "획기적인" 같은 감정적 형용사
- 이모지 사용
- 독자 수준에 맞춰 어려운 용어를 무리하게 풀어쓰기
- 주제 의식에 집착하여 추상적 가치를 강조하기

## 📚 Study (학습)

### 특징
- **개념 설명 중심**: 학습한 개념을 정확하고 체계적으로 설명
- **이해도 향상**: 독자가 쉽게 따라할 수 있도록 명확한 단계나 예시 포함
- **체계적 정리**: 학습 과정이나 정리 내용이 논리적인 흐름을 갖도록

### 작성 규칙

1. **개념 설명의 정확성과 이해도를 높이는 데 집중**

2. **학습 과정이나 정리 내용이 체계적이고 논리적인 흐름을 갖는지 확인**

3. **독자가 쉽게 따라할 수 있도록 명확한 단계나 예시를 포함**

4. **문체**
   - "~다."로 끝나는 건조하고 분석적인 문체
   - 감정 배제

## 📊 Document/Essay (분석/에세이)

### 특징
- **데이터/근거 기반**: 모든 주장은 데이터나 근거에 기반
- **비판적 분석**: 다양한 관점을 제시하고 비판적으로 분석
- **인사이트 도출**: 깊이 있는 인사이트 제공

### 작성 규칙

1. **데이터/근거 기반으로 타당성을 보강**

2. **다양한 관점을 제시하고 비판적인 분석을 통해 깊이 있는 인사이트 제공**

3. **구조**
   - [현상/문제 인식] → [데이터/근거 분석] → [전문가 의견 대조] → [인사이트 도출]

4. **참조 형식**
   - 외부 자료는 [^n] 형식으로 참조
   - 글의 맨 마지막에 ## References 섹션 추가
   - 형식: `[^1]: [문서 제목/웹사이트명](URL) - 간단한 설명`

5. **전문가 의견 인용**
   - blockquote 형식 사용
   - 인용문 아래에 발언 주체 명시
   - 예시:
     ```
     > "발언 내용"
     > — *발언자 이름 (소속)*
     ```

6. **문체**
   - "~다."로 끝나는 건조하고 분석적인 문체
   - 감정 배제

### 금지 사항
- "안녕하세요", "반갑습니다" 같은 인사
- "결론적으로", "요약하자면" 같은 접속사
- "매우", "획기적인" 같은 감정적 형용사
- 이모지 사용

## 🔧 공통 규칙

### 모든 카테고리 공통

1. **언어**: 반드시 한국어(한글)로만 작성
2. **문체**: "~다."로 끝나는 문체 (Daily는 예외적으로 개인적 느낌 허용)
3. **길이**: 최소 1500자 이상 (Study, Dev, Document는 1200자 이상)
4. **이모지**: 절대 사용하지 않음
5. **Front Matter**: 다음 필드를 포함해야 함
   - **시간대**: `date` 필드는 반드시 `+0900` (한국 표준시, KST)을 포함해야 함
     - 예: `date: 2026-01-07 12:40:00 +0900`
   - **permalink**: URL 구조는 `/카테고리/번호/` 형식으로 설정
     - 예: `/dev/1/`, `/daily/1/`, `/study/1/`, `/document/1/`
     - 각 카테고리별로 날짜순으로 번호를 부여 (가장 오래된 글이 1번)

### 참조 및 인용

- 외부 자료: [^n] 형식으로 참조, 마지막에 ## References 섹션
- 전문가 의견: blockquote 형식 사용
- 코드 블록: 마크다운 코드 블록 형식 사용
